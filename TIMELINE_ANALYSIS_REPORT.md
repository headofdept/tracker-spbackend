### 🚨 Критические узкие места

1. **Статус "Ревью" — 59 дней медиана**
   - Это означает, что задача 2 месяца ждёт code review
   - Причины: нехватка time на review, сложный код, недостаток reviewers
   - **Рекомендация:** Добавить SLA на code review (макс 2-3 дня)

2. **"Требуется информация" — 58 дней**
   - Задача зависает в ожидании информации от клиента/PM
   - **Рекомендация:** Auto-close через 2 недели если info не получена

3. **Открытые задачи — 33.5 дней в "Открыт"**
   - Даже новые задачи неделю-полторы ждут начала работы
   - **Рекомендация:** Установить SLA на start (max 5 дней после создания)


### 🤔 Почему "Критичный" и "Средний" выполняются дольше "Блокера"?

Возможные причины:
1. **Неправильная классификация приоритетов** — слово "Критичный" не означает срочность
2. **Объём работы** — критичные задачи может быть крупнее и требуют много времени
3. **Когнитивная нагрузка** — разработчики откладывают сложные "критичные" задачи
4. **Очередь** — много задач с одним приоритетом, ждут очереди на execution

**Рекомендация:** Пересчитать приоритеты на основе срочности (urgency), а не важности (importance).

## 🎯 АНАЛИЗ ГОРЯЧИХ ТОЧЕК

### Задачи-"мастодонты" (> 6 месяцев)

183 задачи выполняются дольше 6 месяцев. Это может означать:

1. **Заброшенные задачи** — никто на них не работает (dead projects)
2. **Очень сложные задачи** — требуют много research и iteration
3. **Блокированные задачи** — ждут external dependency
4. **Неправильно спланированные** — разбита на подзадачи, но parent остаётся открытой

**Рекомендация:** Провести аудит всех задач > 6 месяцев:

## ⚠️ ПРОБЛЕМЫ В ПРОЦЕССАХ ПОСТАВКИ

### Проблема 1: Очень длинный код review (59 дней в "Ревью")

**Симптомы:**
- Задача ждёт approval от reviewers 2 месяца
- Это огромный cost

**Причины (вероятные):**
- Нет культуры code review
- Reviewers перегружены
- Код слишком сложный для быстрого review
- Нет SLA на review

**Решение:**
1. Установить SLA: code review должен быть закрыт за 2-3 рабочих дня
2. Если reviewer не доступен — escalate к lead
3. Требовать smaller commits/PRs (< 400 строк кода)

### Проблема 2: Долгое ожидание start работы (33 дня в "Открыт")

**Симптомы:**
- Задача создана, но в работе ещё месяц

**Причины:**
- Нет приоритизации
- Разработчики заняты другим
- Задача неясна (нет spec)

**Решение:**
1. SLA на start: max 5 дней после создания
2. Если задача не стартована за 5 дней → либо увеличить priority, либо закрыть
3. Требовать acceptance criteria перед start

### Проблема 3: Информационные блокировки (58 дней)

**Симптомы:**
- Задача ждёт info от клиента/PM

**Решение:**
1. Auto-close через 2 недели если info не получена
2. Escalate PM если информация критична
3. Требовать info **перед** созданием задачи

## 📋 ПЛАН ДЕЙСТВИЙ ДЛЯ УСКОРЕНИЯ

### Краткосрочные (1-2 недели)

1. **Установить SLA для каждого статуса:**
   - В работе: 1-2 неделю (есть progress)
   - Ревью: 2-3 дня (critical!)
   - Открыт: 5 дней (нужно начать)
   - Требуется инфо: 2 недели (потом auto-close)

2. **Провести code review audit:**
   - Какие задачи ждут > 2 месяцев в Ревью?
   - Почему reviewers не одобряют?
   - Есть ли bottleneck?

3. **Пересчитать приоритеты:**
   - "Критичный" vs "Блокер" — переходитесь ли вы?
   - Должны ли "Критичные" выполняться дольше?

### Среднесрочные (1-2 месяца)

4. **Меньше и чаще:**
   - Разбить крупные задачи на подзадачи
   - Target: max 2-3 недели на задачу

5. **Улучшить код review культуру:**
   - Обучение коллегам на примерах
   - Pair programming для сложного кода
   - Требовать timely review как KPI

6. **Автоматизировать процессы:**
   - Сигналы в Slack когда task > X дней в одном статусе
   - Автозакрытие "Требуется информация" через 14 дней

### Долгосрочные (квартал)

7. **Переучить команду:**
   - Metering: small features, frequent releases
   - Continuous delivery вместо batch releases

8. **Пересчитать load:**
   - Даниил Сычев перегружен (45% всех задач)
   - Нужно либо нанять, либо распределить иначе
