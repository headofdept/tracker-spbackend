# ⚠️ ИСПРАВЛЕННЫЙ АНАЛИЗ: Время в статусе RC

## 🔴 КРИТИЧНОЕ УТОЧНЕНИЕ

**Предыдущий анализ содержал методологическую ошибку.**

Было рассчитано: `lead_time = updated_at - created_at` для RC задач  
Было получено: медиана 47 дней

**Это неправильно**, потому что:
- `created_at` — когда задача была создана в Tracker
- `updated_at` — когда задача была последний раз изменена
- RC может быть статусом, в который задача попала намного позже создания
- Но мы **не знаем** точно когда она вошла в RC статус

## 🎯 ЧТО НУЖНО НА САМОМ ДЕЛЕ

Нам нужно знать для каждой RC задачи:
1. **Timestamp когда задача вошла в RC статус** (переход Status → RC)
2. **Timestamp когда задача вышла из RC** (переход RC → Done/Resolved)
3. **Разница между этими моментами = время в RC**

## 📊 ТЕКУЩИЕ ДАННЫЕ

CSV экспорт содержит:
```
task_key, created_at, updated_at, status, ...
SPBACKEND-556, 2025-10-31, 2026-06-24, RC, ...
```

**Это НЕ даёт** информацию о:
- Когда задача вошла в RC (могла быть в другом статусе месяцы)
- Сколько дней она провела в других статусах перед RC
- Была ли она раньше в RC и вернулась туда
