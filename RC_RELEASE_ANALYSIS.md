### Проблема 2: Батчинг релизов (акумуляция)

**90% RC задач ждут дольше 7 дней** — это означает, что релизы:
- Выпускаются редко (раз в месяц или реже)
- Аккумулируют много изменений в одном релизе
- Становятся сложнее для QA
- Повышают риск регрессий

**Правильно:** Релизить weekly или bi-weekly.

## 🎯 ПРОБЛЕМЫ И РЕШЕНИЯ

### Проблема A: Задачи мёртвых (мастодонты)

**Что:** 236 дней в RC — это явно заброшенная задача.

**Почему:** 
- Упущена из внимания
- Ждёт чего-то внешнего (не завершилось)
- Технически невозможна к релизу

**Решение:**
```bash
# Audit: для каждой RC задачи
for task in SPBACKEND-556, SPBACKEND-255, ...
  1. Проверить: она реально нужна?
  2. Если НЕТ → откатить или закрыть
  3. Если ДА → выяснить блокер и ускорить
```

### Проблема B: Батчинг релизов

**Что:** Релизятся редко, потому очередь растёт.

**Почему:**
- Страх перед часторелизированием
- Сложный процесс релиза
- Нет CI/CD для релизов

**Решение:**
1. **Перейти на weekly릴리스:** план выпусков (каждый пн)
2. **Ускорить процесс RC→Release:** max 2 дня от RC до Release
3. **Автоматизировать:** CI/CD pipeline для релизов

### Проблема C: Нет SLA на RC

**Что:** Нет правила, сколько дней ждать в RC.

**Почему:**
- Нет владельца релиз-процесса
- Нет мониторинга
- Нет алертов

**Решение:**
```
✅ SLA на RC: max 2 дня

Если задача > 2 дня в RC:
  • День 1-2: ждём QA финализировать
  • День 2: если не готова → откатываем
  • День 3+: эскалируем техлиду
```

## 🔨 ИНСТРУМЕНТЫ И ЧЕКЛИСТЫ

### Чеклист "RC готов к релизу"

Перед переходом из RC в Release:

- [ ] Все баги из Testing закрыты
- [ ] Основной функционал протестирован
- [ ] Performance тесты пройдены (если нужны)
- [ ] Security review завершен (если нужен)
- [ ] Документация обновлена
- [ ] Release notes написаны
- [ ] Rollback план готов

Если не всё готово → **останови релиз и уведоми**

- **RC Owner** — человек, который следит за очередью RC

## 💡 ПОЧЕМУ ЭТО КРИТИЧНО

### Каскадный эффект

```
RC задачи ждут 47 дней
  ↓
Батчинг релизов раз в месяц
  ↓
Одна ошибка в батче → откатываем весь релиз
  ↓
Никто не верит в частые релизы
  ↓
Задачи скапливаются в очереди
  ↓
Lead time растёт (99 дней)
  ↓
Система становится медленнее и нестабильнее
```

### Как это должно быть

```
SLA на RC: 2 дня
  ↓
Еженедельные релизы (вторник)
  ↓
Меньше изменений в одном релизе
  ↓
Проще найти ошибки и откатиться
  ↓
Команда доверяет частым релизам
  ↓
Задачи не скапливаются
  ↓
Lead time → 14 дней (вместо 99)
```
